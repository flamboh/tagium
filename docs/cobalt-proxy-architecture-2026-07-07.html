Cobalt Proxy Architecture Changes

How Tagium should let Cobalt burst without letting one shared-vCPU Machine build an unbounded backlog.

Companion to cobalt-load-test-2026-07-07.html. Generated 2026-07-07.

Decision

Move Cobalt's native request rate limits out of the normal production path, then rely on the wrapper proxy's concurrency gate as the primary backpressure mechanism. 

RATELIMIT_WINDOW=60
RATELIMIT_MAX=1000
TUNNEL_RATELIMIT_WINDOW=60
TUNNEL_RATELIMIT_MAX=2000

These values keep Cobalt's limiter as an abuse backstop while avoiding accidental 429s during healthy bursts. The operational limiter becomes the proxy gate: active work is capped, excess work queues for up to 30s, and overload fails fast with 503.

Native Cobalt Limits

  • Cobalt defaults to 20 resolve requests per 60 seconds and 40 tunnel requests per 60 seconds.
  • The load test hit real 429s at concurrency 4, before the Machine was close to saturation.
  • Native limits are request-rate limits. They do not understand active work, queued work, or whether a request will keep the event loop busy for seconds.

Proxy Concurrency Gate

  • The wrapper gates POST / resolve traffic separately from GET /tunnel traffic.
  • Defaults are 24 active resolve requests and 48 active tunnel requests, with bounded queues of 96 resolve requests and 192 tunnel requests.
  • Queue overflow or timeout returns Cobalt-shaped 503 JSON with error.api.capacity_exceeded.

User-Visible Queue

Tagium already queues playlist downloads and browser-local Cobalt download slots. That app queue is the user-facing layer: it can show position, ETA, retry, and cancel controls before work reaches Cobalt.

The proxy queue is deliberately different. It is a hidden Machine backstop for shared traffic across users and browser sessions. It absorbs short bursts for up to 30s, then returns error.api.capacity_exceeded instead of letting a single Cobalt Machine build a minutes-long backlog.

Production capacity control. It rejects by request count, not by current Machine pressure.
Proxy concurrency gate
Protecting one Fly Machine from too many active resolve/tunnel operations at once.
Long-window abuse prevention. It intentionally permits bursts up to available capacity.

Rate Limit Identity

These Cobalt limits are not per browser user. They are per identity Cobalt observes.

  • For POST /, Cobalt keys rate limits by API key/session key when auth is used; otherwise by the client IP Cobalt sees.
  • For GET /tunnel, Cobalt keys by the client IP it sees.
  • In Tagium production, browsers do not call Cobalt directly. The Tagium server/Worker calls Cobalt, so many browser users can share the same Cobalt-observed caller identity.
  • Without Redis-backed Cobalt rate-limit storage, the counter is local to the Cobalt process/Machine.

Result: the recommended values are Machine-side guardrails, not user quotas. Per-browser fairness belongs in Tagium's own API layer.

Abort Propagation

Abort handling keeps the proxy gate aligned with real client demand. It prevents canceled browser downloads from continuing to consume upstream Cobalt slots after the user has left.

Queued request Client abort cancels the queue entry before any slot opens.
Gate slot Already-gone clients release the slot immediately.
Active upstream Client abort destroys the matching upstream Cobalt request.
Response end Normal upstream completion releases the gate slot.
Response abort/error Upstream aborts and errors also release the slot once.

This directly addresses the load-test collapse mode: at high concurrency, client requests timed out while the Machine kept draining old work for minutes. Aborts make canceled work visible to the proxy and reduce post-timeout backlog.

Mapping To Test Results

  1. The report's first failure was 429, caused by Cobalt's default 20/60s and 40/60s limits. Raising the native limits prevents that false ceiling.
  2. With the native limiter raised during testing, the Machine handled normal bursts and then degraded by accumulating work. The proxy gate turns that degradation into bounded queueing plus fast 503 overload responses.
  3. The test showed memory stability and no Machine crash. That points to scheduler/event-loop/backlog pressure, which is exactly what an active-work gate controls.
  4. The post-wave drain after the highest-concurrency run is the reason abort propagation matters. If the caller is gone, the corresponding Cobalt request should stop competing for the same single Machine.
