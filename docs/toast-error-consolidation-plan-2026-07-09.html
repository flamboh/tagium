Implementation plan · revised 10 July 2026

Keep form errors local. Put system failures in toasts.

Reserve stable validation space beside fields that users can correct. Consolidate download, network, worker, export, and persistence failures behind one user-facing reporting module when there is no honest field-level home. Preserve track-level recovery state when a failure happens after an import enters the queue.

No new dependency Sonner remains the notification surface Motion-safe and accessible URL imports first

Outcome

01

One friendly system explanation

System and download failures map to a stable title and useful next step. Raw Cobalt codes, response bodies, and developer-oriented messages never reach the user.

02

Local validation without layout shift

Required fields reserve their error row before it is needed. Errors appear close to the field, without growing a dialog or moving adjacent controls.

03

Persistent state where recovery lives

Queued-track failures still remain on the track and queue because that is where retry is available. Toast consolidation must not erase recoverable state.

Recommended product decision

Use a short visual shake, not device vibration. A CSS response is predictable across desktop and mobile, needs no permission, and can cleanly respect reduced-motion preferences.

Current state

The repository has the right primitives, but error ownership and layout behavior are split. Form validation shifts layout, while system failures are duplicated, raw, or invisible to the user.

Two URL-entry implementations

LandingScreen.tsx and AudioDownloader.tsx each own URL, loading, inline error state, and nearly identical catch logic. Both render an aria-live="polite" paragraph below the input.

Partial toast routing

downloadErrorMessage.ts only toasts capacity, service-unavailable, timeout, and rate-limit categories. Ordinary failures fall through to raw Error.message and appear inline.

Delayed failures use a different seam

Single-track imports are accepted into the queue immediately. Later errors flow through markDownloadError, update TagiumFile.downloadError, and are rendered in TrackMetadataEditor.

The global notification host is ready

App.tsx already mounts one rich-color, closeable Sonner toaster. sonner.tsx and index.css already define duration and desktop sizing, so this work does not need another notification library.

Album validation grows the dialog

AlbumMetadataDialog.tsx conditionally inserts title and artist error paragraphs. The surrounding grid already has room, but the missing reserved rows let the dialog height change when validation appears.

Filename has no validation state

The track header already has an error-reporting area below the filename, but an empty manual filename is not marked invalid or reported there. Other malformed characters can be normalized quietly with the existing filename sanitizer.

Important timing constraint

The target design keeps one media-entry DOM node mounted while it moves from landing to editor layout. Even so, a delayed failure must not mark that field after the user has started another entry. Pre-queue validation belongs to the current field; post-queue failure belongs to the track, queue, and system-failure toast.

Do not apply “toast everything” literally

Toasts are for failures without a specific corrective field. Required-field errors, failed tracks, queue progress, and retry affordances remain in context. Consolidation means clear ownership, not routing every error through one visual surface.

Field validation and stable layout

Validation that the user can fix in a text field stays next to that field. The layout reserves the message row at rest, so showing an error changes color and text—not geometry.

Album dialog

  • Reserve one fixed-height error row below album title and artist.
  • Keep the row present whether or not its message is visible.
  • Connect each input and message with aria-describedby.
  • Keep aria-invalid on the field after a failed submit.
  • Do not use a tooltip for required-field errors.

Track filename

  • Keep a reserved status area directly below the filename/title header.
  • Show “filename is required” when the sanitized filename is empty.
  • Mark the editable filename input invalid and associate it with the message.
  • When filename sync is enabled, validate the filename derived from title.
  • Silently sanitize other malformed filename characters.
Tooltip rule

Use tooltips for explanatory help or disabled-control reasons. Do not hide actionable validation in hover/focus UI: reserved text is visible on touch, easier to scan, and provides a stable accessible description.

Interaction details

  • Keep the rejected URL in place so it can be corrected.
  • Keep focus where the user left it; do not steal focus for the toast.
  • Set aria-invalid="true" during the error state.
  • Clear the destructive state as soon as the user edits or resubmits.
  • Use a monotonically increasing feedback key so repeated failures replay.

Accessibility and motion

  • Local validation is associated with its field and announced once.
  • A non-motion red border/ring still communicates the source visually.
  • prefers-reduced-motion: reduce disables translation.
  • Color is not the only signal: message text and invalid semantics remain.
  • System toasts use Sonner’s live region without a duplicate local message.

Target architecture

Keep ordinary form validation inside each form module. Put technical failure recognition, safe public copy, notification policy, and deduplication behind one deep system-failure reporting module. Callers should never decide system copy by inspecting raw strings.

Technical sources Cobalt, fetch, worker, decoding, export, metadata write, storage.
→
System-failure reporting module Classify, redact, phrase, dedupe, and route technical failures.
→
Surfaces Toast, durable track summary, queue recovery, analytics category.

Pre-queue rejection

The persistent media-entry module leaves the URL intact. Invalid syntax and unsupported input stay local to the field; network or server failures go through the system reporter. Only field failures increment the input feedback key.

Post-queue settlement

The download controller/track boundary stores only friendly track copy and requests a deduped toast. The input is no longer involved. Set failures use one aggregate toast per category rather than one notification per track.

System presentation

The reporting module derives a structured value rather than leaking a loose message:

code, title, description, trackDescription, retryable, and dedupeKey.

Keep technical causes available to logs/analytics categorization, but never place them in toast descriptions or persisted user-facing fields.

Notification contract

Replace the current category-only notifyDownloadError behavior with a notifier that accepts the structured presentation and always has a safe fallback.

Retain the existing 15-second category debounce for shared service failures. Give direct system failures a stable per-attempt policy so a user action gets feedback without creating a playlist toast storm. Field validation bypasses this notifier.

Incremental seam

Replace downloadErrorMessage.ts with the deep systemFailure.ts reporter. Keep field validation in the album, track, and media-entry modules; do not create a second generic form-error abstraction.

Copy audit during implementation

Capture the actual Cobalt error-code fixtures already exercised by the API and local processing tests. Expand the mapping from evidence; do not guess every provider code in advance. Unrecognized values must land on the safe fallback.

Execution phases

Phase 0

Lock field and system routes with tests

Assert that album errors do not change dialog geometry, empty sanitized filenames use the reserved track-header spot, malformed characters normalize quietly, URL syntax stays local, and transient/unknown system failures use safe toast copy.

Deliverable: failing tests express stable layout, local field ownership, friendly system copy, and no raw message leakage.

Phase 1

Create one deep system-failure reporting module

Refactor the current category logic into a total function from unknown + context to safe structured copy. Preserve analytics error categorization as a separate concern and add table-driven unit coverage for every public category.

Deliverable: no system/download caller needs instanceof Error or string inspection to decide user-facing text; forms keep their own validation rules.

Phase 2

Stabilize local form validation

Reserve album title and artist message rows so the dialog never grows on validation. Keep the track header’s existing status area mounted and add empty sanitized filename validation there. Associate messages with fields and keep sanitation quiet.

Deliverable: album and track form errors remain close to their fields without layout shift or tooltip-only validation.

Phase 3

Render one persistent media-entry module

Keep one media-entry DOM node mounted outside the landing/editor conditional and move it by layout mode. It owns URL state, loading, local validation, the reserved message row, and repeatable invalid feedback. Delete the second URL-entry implementation.

Deliverable: value, focus, loading, and feedback survive the first-track transition; one interface and one test surface cover both layouts.

Phase 4

Add motion-safe input styling

Add a scoped keyframe for a restrained horizontal shake and red ring. Replay it via a keyed wrapper or forced animation-name toggle; never rely on a boolean that fails to restart. Under reduced motion, keep the ring/color transition and remove translation.

Deliverable: clear contextual feedback at both breakpoints with no layout shift.

Phase 5

Route queued failures through the system reporter

Change markDownloadError to persist only friendly track copy and request a deduped toast. Keep track retry and queue summary behavior. Ensure playlist failures aggregate by public category rather than creating one toast for every track.

Deliverable: immediate and delayed failures use the same language without targeting stale URL controls.

Phase 6

Audit the next error surfaces without expanding the first change

Inventory cover-art, metadata-save, upload-parse, and export errors against the same display rules. Keep inline form validation and persistent recovery state. Convert only action-level explanations that are currently raw, duplicated, or easy to miss.

Deliverable: a small follow-up list, not an opportunistic app-wide refactor in the URL error change.

✓No field error is duplicated in a system toast.
✓Every URL submission failure reaches its correct local or system surface.
✓Raw response bodies and upstream codes never appear in UI.
✓The rejected URL remains editable and focus is not stolen.
✓Album validation never changes dialog height.
✓Empty sanitized filenames use the reserved track-header status area.
✓Malformed filename characters are sanitized without an error.
✓The input pulse replays for consecutive failed attempts.
✓Reduced motion removes all shake translation.
✓Screen readers announce one useful error, not duplicate live regions.
✓Delayed failures remain attached to retryable tracks.
✓Playlist failures do not create a toast storm.
✓Successful resubmission clears invalid styling and URL value.

Scope control

The first implementation covers stable album validation, empty filename validation, one persistent media entry, set resolution, queued URL downloads, and core system reporting. It does not create a generic form-error framework or redesign queue UI, analytics taxonomy, or the Effect error hierarchy.
