Cobalt Fly Machine — Load Test Results

Where the current single-Machine Cobalt setup actually breaks, measured directly against a disposable clone of production infrastructure — not estimated.

2026-07-07 · target: tagium-cobalt-loadtest.fly.dev (disposable clone of tagium-cobalt) · region: lax · torn down after testing

TL;DR

  1. The real ceiling today isn't Fly hardware — it's Cobalt's own default rate limiter. Cobalt ships with a built-in cap of 20 resolve requests/60s and 40 tunnel requests/60s, keyed per client IP, unrelated to Machine size. We hit real 429s at concurrency as low as 4.
  2. With that limiter raised, the single shared-cpu-1x / 1GB Machine handles roughly 30–64 concurrent active downloads comfortably, starts showing cracks around 96–150, and collapses past ~200 (60% error rate, most requests hitting our 120s client timeout).
  3. The failure mode is queueing, not crashing. Memory stayed healthy throughout; the Machine never restarted. It just fell behind and kept draining the backlog for minutes after the client gave up.
  4. Total cost of this entire test session: about $0.05.

Methodology

  • Cloned fly.cobalt.toml to a disposable app, tagium-cobalt-loadtest, scaled to exactly one Machine to match production's actual footprint (checked live — production tagium-cobalt also runs exactly one Machine today).
  • Overrode API_URL at deploy time so Cobalt's generated tunnel URLs pointed back at the clone, never at production. Verified this before any load test ran.
  • Hit the Cobalt Fly Machine directly, bypassing Tagium's own Cloudflare Worker layer, to isolate the Machine's own ceiling from the (already known to be non-bottleneck) Worker layer.
  • Used scripts/load-test-cobalt.ts with the curated scripts/load-test-urls.txt list — 7 real, openly-licensed source URLs (Big Buck Bunny, "Me at the zoo", NoCopyrightSounds tracks on YouTube and SoundCloud) cycled across requests at increasing concurrency.
  • Sampled /proc/loadavg and /proc/meminfo on the Machine via flyctl machine exec after each wave, and cross-referenced the proxy's own structured request logs (cobalt-machine-proxy.mjs) via flyctl logs.
  • Got a second opinion from GPT-5.5 (via codex exec) on the raw results and methodology before writing conclusions — see below.
  • Destroyed the disposable app immediately after testing finished.

Finding 1 — Cobalt's default rate limiter is the real bottleneck highest priority

At concurrency 4, two of twelve requests failed with HTTP 429. Response headers gave it away immediately:

ratelimit-policy: 20;w=60 — 20 requests per 60 seconds, per client IP.

This comes straight from Cobalt's default config (repos/cobalt/api/src/core/env.js), via express-rate-limit:

  • RATELIMIT_MAX = 20 per RATELIMIT_WINDOW = 60s on POST / (resolving a download plan)
  • TUNNEL_RATELIMIT_MAX = 40 per TUNNEL_RATELIMIT_WINDOW = 60s on GET /tunnel
  • Keyed by a hashed client IP address, stored in-memory (no REDIS_URL is configured, so each Machine tracks its own counter independently)
  • Can be raised via those four env vars, or bypassed per-caller with a Cobalt API key that has a custom limit (including "unlimited") configured in Cobalt's key file

Why this matters more than it looks: the browser never calls Cobalt directly — Tagium's own Cloudflare Worker does, on every user's behalf. That means every simultaneous Tagium user's resolve calls funnel through Cobalt's rate limiter keyed by whatever IP the Worker's outbound request presents. If that's a single IP or a small pool (plausible for a serverless platform's egress), this 20/60s budget is likely shared across your entire user base at once, not per user. We couldn't confirm this from outside — it needs a look at what IP Cobalt actually sees from production traffic (see Open Questions).

We disabled this limiter on the clone (env vars set to 100000/60s) to isolate the Machine's actual hardware ceiling, which is Finding 2.

What kind of failure is this?

Not a crash. Across the entire test — including the 220-concurrency wave that pushed 60% of requests to failure — the Machine's state never changed from started, and free memory only dropped from ~700MB to ~570MB of 985MB total (never close to OOM). The 1-minute load average, on a Machine with exactly one shared vCPU, climbed steadily and predictably with concurrency:

That's the signature of CPU/event-loop saturation on a single-threaded Node process, not memory or network exhaustion. After the 220-concurrency wave ended (client-side), server logs kept logging successful 200 completions for minutes afterward — one request finished in 204449ms (204 seconds) against our 120s client timeout. The Machine was still grinding through the backlog long after every client had already given up and moved on. It eventually drained fully and auto-suspended cleanly on its own — no manual intervention needed.

Second opinion — GPT-5.5, via codex exec

Asked to critique the methodology and conclusions independently, without reading this write-up, from the raw numbers alone.

On the interpretation

"Default Cobalt limits imply roughly 20 resolve/min/IP and 40 tunnel/min/IP. If Cloudflare Worker egress collapses many users onto one or a few IPs, Tagium hits this before hardware matters... I would call 30–60 active downloads acceptable, 100+ overloaded, 200+ unusable."

On the failure signature

"Load rising without memory pressure strongly points away from OOM and toward CPU/event-loop/scheduler saturation... Cobalt may do synchronous or CPU-heavy source resolution work: signature deciphering, crypto, parsing, redirects... The strongest evidence for saturation is the post-timeout drain: the machine did not die; it accumulated more work than one CPU/event loop could retire within client deadlines."

Biggest blind spot flagged

"The test bypassed the real production path... What IP does Cobalt actually see in production for POST /? ... Does Cloudflare Worker egress use one IP, a small regional pool, or broad per-colo diversity? ... Only 7 source URLs — that may under-sample source variability."

Recommended fix order

"Fix rate limiting intentionally first... Measure production caller identity... Add queue/backpressure... Set timeouts and abort propagation... Scale before tuning internals [only] for this workload... Re-test through the real app path."

Recommendations, in order

  1. Configure Cobalt's rate limits intentionally. Either raise RATELIMIT_MAX/RATELIMIT_WINDOW/TUNNEL_RATELIMIT_MAX/TUNNEL_RATELIMIT_WINDOW to a value chosen on purpose, or configure a Cobalt API key with an explicit limit and have Tagium authenticate with it. This is a five-minute, zero-cost fix and is almost certainly the single highest-leverage change available right now.
  2. Find out what IP Cobalt actually sees from production. Log the client IP (or hashed rate-limit key) Cobalt observes on real Tagium Worker traffic. This determines whether the current 20/60s default is a per-user annoyance or a whole-app-wide ceiling.
  3. Add an explicit concurrency cap in front of Cobalt, tuned to roughly 30–60 in-flight downloads (the zone that stayed fast and error-free here), with a friendly "please wait" / queued response beyond that — rather than letting requests pile up past the point where Cobalt itself falls over.
  4. Propagate client aborts upstream. When a browser gives up waiting, cancel the corresponding Cobalt request instead of letting it keep consuming the Machine's one CPU core in the background — this is exactly what caused the multi-minute backlog drain after the 220-concurrency wave.
  5. Don't reach for multi-Machine scaling yet. The machine-affinity plumbing already exists in this codebase if it's ever needed, but real sustained traffic would need to approach the 30–60 concurrent range before that's the right lever to pull. Fixing #1–#4 buys much more headroom for much less effort.
  6. Re-run this test through the real path (browser → Tagium Worker → Cobalt) once the above changes land, to confirm the same knee shows up and that Tagium's own per-IP limiter is doing its job before requests ever reach Cobalt.

Open questions / methodology caveats

  • This test bypassed Tagium's Cloudflare Worker entirely to isolate the Cobalt Machine's own ceiling. Real end-to-end latency will be higher, and the Worker's own 60/min per-IP limiter and same-origin check add behavior this test didn't exercise.
  • Only 7 source URLs were used, all openly licensed and chosen for repeatability. This was deliberate (to avoid provider rate-limiting or ban risk), but it doesn't rule out YouTube/SoundCloud-side throttling showing up differently under more diverse, higher-volume, sustained real traffic.
  • The exact response bodies for the http-500 errors at concurrency 96/150 weren't captured — only status code and byte count. scripts/load-test-cobalt.ts currently doesn't retain response bodies on non-2xx tunnel/resolve failures; worth adding if this test gets re-run.
  • Single test session, single region (lax), no sustained soak test (this was concurrency ramps of seconds-to-minutes each, not sustained load over hours).
  • This connects back to the earlier scaling-plan review: the plan proposed an async job queue + R2 cache to solve Cobalt's process-local tunnel state. That's still true as a scaling concern for later, but it's not the current bottleneck — the rate limiter is a much cheaper and more urgent fix than any of that infrastructure work.

Cost & cleanup

  • Total data transferred across all waves: ~2GB (≈ $0.04 at Fly's ~$0.02/GB egress).
  • Machine compute time: ~15 minutes total across the whole session at shared-cpu-1x/1GB ($0.0000022/s) ≈ $0.002.
  • Total: about $0.05.
  • tagium-cobalt-loadtest was fully destroyed (flyctl apps destroy) immediately after testing. Production tagium-cobalt was never touched — confirmed untouched throughout via its own machine list and suspended state.

Addendum — cover art doubles tunnel load, and the queue works

Follow-up testing done the same day, after a real question surfaced a gap in the original methodology.

The original test undercounted real downloads

The first pass of this test only fetched tunnel[0] (audio) per simulated download. Real downloads fetch every tunnel Cobalt returns concurrently — audio plus cover art, when present (see components/audio/localAudioProcessor.ts's Effect.all([audio, cover], { concurrency: 2 })). Cover art is the norm for music tracks, not the exception. Fixed the script and re-ran the same concurrency levels with the corrected behavior.
