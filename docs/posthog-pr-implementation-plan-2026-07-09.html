Tagium · PostHog PR #65

Make analytics truthful, isolated, and lightweight

A concrete plan to rebuild the wizard-generated instrumentation around a typed module, emit outcomes from real workflow seams, activate PostHog only in the production deploy, and restack the result safely on the stability work in PR #66.

Prepared 2026-07-09 No session replay #66 must merge first Graphite child PR

Outcome

PR #65 will become a narrow analytics change on top of PR #66. UI modules will report domain facts through one typed interface; the analytics implementation will own SDK loading, event names, common properties, provider/error normalization, payload redaction, and environment gating.

Correct

Events describe observed outcomes

Upload, import, export, cancellation, and retry events fire after the corresponding workflow milestone—not when a button is clicked or work is merely queued.

Isolated

Only production can send

Local, test, E2E, and Cloudflare preview builds have no key and use the no-op adapter. Production additionally requires an explicit production build environment.

Proportional

The SDK leaves the startup path

PostHog loads during idle time into its own chunk. A bounded in-memory queue keeps early domain events until the client is ready.

Hard merge constraint

Do not merge PR #65 before PR #66. Both change components/audio/audioTagger.tsx, especially import and removal flows; #66’s confirmation, session-safety, memory, and deploy behavior is the baseline to preserve.

Deep implementation

  • Provider and stable error-code mapping.
  • event_version, deploy_env, and release SHA.
  • Production-only initialization and no-op adapter.
  • Idle-time dynamic import plus bounded early-event queue.
  • before_send denylist/redaction as a final guard.
  • Explicit PostHog privacy and collection configuration.

Accurate import aggregation

The existing playlist controller can merge newly enqueued tracks into an active run, so one terminal queue snapshot cannot reliably describe one URL or SoundCloud set. Add a small ImportLifecycleTracker keyed by an opaque, in-memory operation ID. The ID travels with managed queue tracks but is never sent to PostHog. The tracker receives per-track terminal outcomes from the controller and emits exactly one final event when every expected track is completed, failed, or canceled.

Controller change

Add a generic lifecycle callback to the controller dependencies for track completion, failure, and cancellation. Keep the controller analytics-agnostic: it reports typed workflow facts, and its existing tests prove ordering and exactly-once behavior.

Tracker change

Start an operation after input validation; record SoundCloud set resolution; attach the local operation ID to planned tracks; aggregate terminal outcomes; classify the final result as completed, partial, failed, or canceled.

Event contract and emission points

Event names describe facts Tagium can actually observe. Property sets are fixed and avoid URLs, names, metadata, response bodies, and arbitrary exception text.

Event Emit when Allowed event properties Implementation seam
audio_upload_completed Duplicate filtering and parsing have both finished. requested_count, accepted_count, duplicate_count, parse_rejected_count, target_kind handleAudioUpload result, after parseUploads
import_started A non-empty, valid import is accepted for resolution/planning. provider, import_kind handleUrlImport before resolver/controller work
import_resolved A SoundCloud set resolves and a track plan exists. provider, import_kind, resolved_count, has_cover Immediately after resolveSoundCloudSet
import_finished Every track in the operation is terminal, or resolution fails before queueing. provider, import_kind, outcome, total_count, completed_count, failed_count, canceled_count, optional error_code, duration_ms Import tracker fed by controller terminal callbacks
import_retry_started The controller accepts one or more failed/canceled tracks for retry. provider, retry_count, previous_failed_count, previous_canceled_count Controller enqueue result, not the retry button handler
import_cancel_requested An active controller run actually transitions to canceled. total_count, completed_count, active_count, pending_count Controller cancel transition
export_started Tagium begins metadata writing / ZIP preparation. export_kind, track_count, optional album_count Track, album, or library export workflow
export_prepared The Blob is ready and Tagium has triggered the browser download. export_kind, track_count, optional album_count, size_bucket After Blob creation and downloadBlob invocation
export_failed Metadata writing or archive preparation fails. export_kind, error_code Export workflow catch path
album_created The album state update succeeds. seed_track_count, has_cover After createAlbumFromTracks is applied
album_edited The album state update succeeds. track_count, has_cover After updateAlbumMetadata is applied
settings_changed Settings persist successfully and differ from the previous value. Only allowlisted booleans and audio_bitrate handleSettingsChange after save
tracks_removed The #66 confirmation is accepted and removal state is applied. track_count, source_mix removeFiles, never the request/cancel path
Export wording is intentionally conservative

The browser can prove that Tagium prepared a file and triggered an anchor download. It cannot prove that the user kept or saved the file, so no event is named *_downloaded.

Privacy and schema guardrails

Normalization before capture

  • Provider is exactly youtube, soundcloud, or other; hostname is never sent.
  • Errors map to stable codes such as capacity, rate_limited, service_unavailable, timeout, resolve_failed, parse_failed, metadata_write_failed, and unknown.
  • Sizes use coarse buckets where useful, not raw filenames or metadata.
  • Operation IDs stay in memory and are not analytics properties.

Final before_send guard

  • Drop unknown custom event names.
  • Drop properties outside the event-specific allowlist.
  • For $pageview and $autocapture, remove SDK location and referrer properties and replace them with an allowlisted app_view enum; Tagium does not need raw browser locations for product analytics.
  • Reject keys associated with URLs, filenames, titles, artists, albums, artwork, raw messages, response bodies, or tunnel identifiers.
  • Return null rather than sending an event that cannot be made safe.

Explicit PostHog settings

{
  defaults: "2026-05-30",
  capture_pageview: "history_change",
  capture_pageleave: true,
  autocapture: {
    dom_event_allowlist: ["click", "submit"],
    element_allowlist: ["button", "form"],
  },
  mask_all_text: true,
  mask_all_element_attributes: true,
  disable_session_recording: true,
  enable_heatmaps: false,
  disable_surveys: true,
  person_profiles: "identified_only",
  before_send: redactAndValidateEvent,
}

This chooses pageviews plus narrow interaction autocapture, with text and element attributes masked. It deliberately excludes session replay, heatmaps, surveys, console logs, copied text, and broad form/input capture. The exact options will be locked by adapter tests against the installed SDK types. See the PostHog JavaScript configuration reference.

Never capture

Source URLs, browser locations/referrers, hostnames, filenames, track/album/artist values, artwork bytes or URLs, upstream response bodies, arbitrary exception messages, tunnel IDs, or the PostHog project token in documentation/tests.

Bundle decision framework

Default choice

Lazy full SDK in its own chunk

Schedule import("posthog-js") after first paint / during idle time and buffer early events in memory. This removes the reported ~73 kB gzip delta from the initial client chunk without recreating SDK delivery semantics.

Benchmark

Try dist/module.slim

Adopt it only if the installed package exports it cleanly, TypeScript resolves it, and it supports custom capture, pageviews, the selected limited autocapture, masking, and before_send. Otherwise keep the lazy full SDK.

Decision rule

Prefer the smallest solution that retains SDK identity, batching, retries, unload delivery, and opt-out behavior. A CDN snippet merely relocates bytes; a bespoke capture transport is out of scope unless measured SDK startup cost remains unacceptable after code splitting.

Variant Measure Accept when
#66 baseline Initial chunk min/gzip; cold first paint Reference only
Lazy full SDK Initial chunk, lazy chunk, cold first paint, idle request timing Initial path is near baseline and all required features work
Lazy slim SDK Same metrics plus functional config tests Smaller and feature-equivalent; no type/export workaround
